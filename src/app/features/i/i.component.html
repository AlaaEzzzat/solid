Interface Segregation Principle (ISP):

Clients should not be forced to depend on interfaces they do not use. This means that you should not include methods in an interface that are not relevant to all its implementers.

Example:

Suppose we have an interface called Animal which has several methods, including eat(), sleep(), and makeSound(). We also have two classes that implement the Animal interface: Dog and Cat. However, Dog does not need the makeSound() method, while Cat does not need the sleep() method.

Notice that the Animal interface has been separated into smaller interfaces, each representing a specific behavior. Dog only implements the eat() method, while Cat implements both eat() and makeSound().

According to the ISP, clients should not be forced to depend on methods they do not use. By separating the Animal interface into smaller, more specialized interfaces, we allow clients to depend only on the methods they need, making our code more modular, flexible, and maintainable.
